# Dev_20260312_LoveType_MVP_API_Spec_v1.txt

서비스명
LoveType

문서명
MVP API 명세서

버전
v1.0

작성일
2026-03-12

목적
LoveType MVP의 프론트엔드와 백엔드 간 통신 규격을 정의한다.
MVP 범위는 오늘의 이야기 / 연애 이야기 2개 카테고리, 소울카드, 3장 카드 리딩, 저장/공유, 사용 제한까지를 포함한다.

기본 원칙
- 구조는 정식 서비스 기준으로 잡고 기능은 MVP 범위에 맞춘다
- 응답 포맷은 일관된 JSON 구조 사용
- 인증은 구글 로그인 기반 토큰 검증 후 자체 세션 또는 JWT 사용 가능
- 리딩 생성은 백엔드에서 JSON 조합 후 LLM 1회 호출
- 앱은 최대한 가볍게, 백엔드는 저장/제한/검증/중계 중심

================================================================
01. 공통 규칙
================================================================

Base URL 예시
- https://api.lovetype.app/v1

Content-Type
- application/json; charset=utf-8

인증 헤더
- Authorization: Bearer {access_token}

응답 기본 구조
성공
{
  "success": true,
  "data": {},
  "meta": {}
}

실패
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "사용자에게 보여줄 수 있는 문장"
  }
}

공통 에러 코드 예시
- UNAUTHORIZED
- FORBIDDEN
- INVALID_REQUEST
- NOT_FOUND
- RATE_LIMITED
- READING_LOCKED
- PAYMENT_REQUIRED
- INTERNAL_ERROR

시간 형식
- ISO 8601 UTC 저장
- 프론트 표시 시 KST 변환

02. 인증 API
================================================================

2-1. 구글 로그인 토큰 교환

POST /auth/google

설명
프론트에서 받은 구글 id_token 또는 access_token을 백엔드에 전달하고
백엔드가 사용자 생성/조회 후 서비스용 access token 발급

Request
{
  "id_token": "google_id_token"
}

Response
{
  "success": true,
  "data": {
    "access_token": "lovetype_jwt_or_session_token",
    "token_type": "Bearer",
    "user": {
      "id": 1,
      "nickname": "인블루",
      "email": "user@example.com",
      "is_profile_completed": true
    }
  },
  "meta": {}
}

비고
- 최초 로그인 시 users 생성
- profile_completed 여부에 따라 온보딩 분기

2-2. 내 세션 확인

GET /auth/me

설명
현재 로그인 사용자 기본 정보 확인

Response
{

    "user": {
      "id": 1,

08. 카드 셔플 / 리딩 생성 API
================================================================

8-1. 셔플용 카드 후보 조회 (선택)

GET /reading/shuffle-seed?category_key=love&deck_theme_key=default_normal

설명
필수는 아니지만 프론트에서 셔플 연출용 난수 시드나 카드 후보군을 받을 때 사용 가능

Response
{
  "success": true,
  "data": {
    "shuffle_seed": "a8f1c0d9",
    "deck_theme_key": "default_normal"
  },
  "meta": {}
}

비고
- MVP에서는 프론트 로컬 셔플로 대체 가능
- 실제 저장은 final generate에서만 발생

8-2. 리딩 생성

POST /reading/generate

설명
MVP 핵심 API
프론트가 선택한 카테고리, 태그, 카드 3장, 덱 정보를 전달하면
백엔드가 payload 조합 후 LLM 1회 호출하여 리딩 결과 생성 및 저장

Request
{
  "category_key": "love",
  "focus_key": "reunion",
  "deck_theme_key": "default_normal",
  "cards": [
    { "card_key": "the_star", "is_reversed": false },
    { "card_key": "the_moon", "is_reversed": false },
    { "card_key": "the_sun", "is_reversed": true }
  ]
}

서버 처리 순서
1. 인증 사용자 확인
2. profile_completed 확인
3. usage 제한 확인
4. category / focus / deck 유효성 검증
5. profile + soul_card + json 리소스 조합
6. final payload 생성
7. LLM 1회 호출
8. readings 저장
9. reading_cards 3장 저장
10. usage_limits 업데이트

16. 프론트 구현 시 주의점
================================================================

- /reading/generate 호출 전 /usage/check 선확인
- 카드 셔플 연출은 프론트 처리 가능, 저장은 generate 이후만
- 소울카드는 홈/결과/히스토리 상단에서 공통 사용
- 리딩 결과는 저장용 텍스트와 공유용 텍스트를 분리 사용
- deck_theme_key는 항상 명시적으로 전달
- profile_completed false 상태에서는 리딩 관련 API 제한

================================================================
17. 백엔드 구현 시 주의점
================================================================

- 인증 토큰 검증 실패 시 401 고정
- 카테고리/태그/덱은 마스터 테이블 기준 검증
- payload_json은 readings에 그대로 저장
- LLM 호출 실패 시 readings 미생성 또는 failed 상태 별도 처리 고려
- usage 제한은 generate 직전 재확인
- same category 1시간 락은 서버 시간이 기준
- 응답 메시지는 사용자 친화적으로 유지

================================================================
18. 최종 한 줄
================================================================

LoveType MVP API는
개인화 프로필, 소울카드, 2개 카테고리 리딩, 히스토리, 덱 선택, 사용 제한을
얇고 확장 가능한 방식으로 연결하는 백엔드 계약 문서다.
