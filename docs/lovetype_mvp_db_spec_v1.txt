서비스명
LoveType

문서명
MVP DB 명세서

버전
v1.0

작성일
2026-03-12

목적
LoveType MVP의 사용자 정보, 소울카드, 리딩 히스토리, 사용 제한, 결제 확장 구조를 위한
기본 DB 스키마를 정의한다.

기본 원칙
- MVP는 구조를 탄탄하게 잡고 기능은 최소 범위로 구현
- today / love 카테고리 2개만 우선 운영
- JSON 데이터셋은 앱/백엔드 리소스로 관리하고 DB에는 실행 결과와 사용자 상태 중심으로 저장
- 추후 구독, 덱 판매, 테마 확장에 대응 가능한 구조 유지
- DB는 PostgreSQL 기준으로 설계

02. 공통 규칙
================================================================

기본 키
- 모든 기본 PK는 bigint 또는 uuid 가능
- MVP에서는 bigint auto increment 기준으로 작성
- 외부 로그인 uid는 별도 unique 컬럼 보관

시간 컬럼
- created_at
- updated_at
- 필요 시 deleted_at
- timezone은 KST 기준 표시하되 DB는 UTC 저장 권장

문자열 규칙
- category_key, theme_key, mbti 등은 소문자/정규화된 값 사용
- 표시용 한글명은 별도 label 또는 name_kr 컬럼 사용

Boolean 규칙
- is_active
- is_premium
- is_reversed
- is_refunded
등 명확한 상태 컬럼 사용

================================================================

설명
로그인 기반 사용자 기본 계정 테이블

컬럼

- 닉네임은 users와 user_profiles 중복 저장 가능하지만
  표시 속도와 관리 편의를 위해 users에 기본 저장

10. usage_limits
================================================================

설명
사용자별 사용 제한 관리
무료 1일 1회, 유료 동일 카테고리 1시간 제한 처리용

컬럼
- id                      bigserial pk
- user_id                 bigint not null references users(id)
- category_id             bigint not null references reading_categories(id)
- free_used_date          date null
- last_reading_at         timestamptz null
- next_available_at       timestamptz null
- free_daily_count        integer not null default 0
- paid_count_today        integer not null default 0
- created_at              timestamptz not null default now()
- updated_at              timestamptz not null default now()

인덱스
- unique(user_id, category_id)
- index(user_id)
- index(next_available_at)

운영 규칙
- 무료 사용자는 하루 1회
- 유료 사용자는 같은 category 기준 1시간 이내 재생성 불가
- 차후 카테고리별 별도 정책 확장 가능

21. SQL 작성 시 주의점
================================================================

- payload_json은 jsonb 사용
- reading_cards는 reading_id on delete cascade 권장
- usage_limits는 user_id + category_id unique로 고정
- user_profiles는 user_id unique로 1:1 보장
- created_at / updated_at 자동 처리 트리거 권장
- nickname, soul_summary 등 표시용 캐시 데이터는 조회 속도상 허용
- 민감정보 최소화 원칙 유지
- birth_date는 필요한 범위 외 노출 금지

================================================================
22. 최종 한 줄
================================================================

LoveType MVP DB는
사용자 정체성, 리딩 결과, 사용 제한, 카드 덱 상태를 중심으로 설계하고
리딩 의미 체계는 JSON 리소스에서 관리하는
얇고 확장 가능한 구조를 목표로 한다.
