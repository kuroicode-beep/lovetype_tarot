# LoveType Tarot — 개선 의견

**성격:** 코드 검수·기능 정의를 바탕으로 한 제안 목록 (우선순위는 프로젝트 상황에 따라 조정)  
**작성일:** 2026-03-27

---

## 1. 제품·동작 일관성

1. **서버 쿨타임 운영 검증**  
   앱은 서버 `tarot/cooltime`을 단일 진실 원천(SSOT)으로 사용하도록 정리되었다. 다음 단계는 실제 Railway 응답 필드와 앱 파서(`is_available`, `next_available_at`, `remaining_seconds`)가 운영 데이터에서 일치하는지 기기 QA로 확인하는 것이다.

2. **어드민 덱 플래그와 앱**  
   Railway 어드민에서 덱 ON/OFF를 저장해도, 앱은 현재 로컬 규칙(고대비·구독)으로만 덱을 고른다. 원격 정책을 반영하려면 **공개 설정 API + 앱 캐시**가 필요하다.

3. **포인트 차감 타이밍**  
   결과 로딩 성공 후 차감 등 현재 플로우가 의도와 맞는지(네트워크 실패·재시도 시 중복 차감 방지) 한 번 더 점검할 가치가 있다.

---

## 2. 기술 부채·구조

1. **라우팅**  
   일부는 named route, 일부는 `push`만 사용. 화면이 늘어나면 **통일된 라우터**와 인자 타입 안전성이 유지보수에 유리하다.

2. **의존성 주입**  
   싱글톤 직접 참조는 테스트 시 mock이 어렵다. 규모가 커지면 **인터페이스 + 주입** 또는 최소한 `ApiService`에 대한 테스트 더블 진입점을 마련한다.

3. **에러 UX**  
   API 실패 시 스낵바/다이얼로그 패턴을 화면별로 흩어 두지 말고, 공통 위젯이나 mixin으로 정리하면 일관된 메시지·재시도 동작을 줄 수 있다.

---

## 3. 품질 보증

1. **테스트**  
   `widget_test` 외에 **API 모킹** 기반의 `ApiService`·`PaymentService` 단위 테스트, 주요 플로우 위젯 테스트를 단계적으로 추가한다.

2. **분석 옵션**  
   `flutter analyze` 유지 + CI에서 동일 명령 실행으로 회귀 방지.

## 4. 빌드·배포

1. **APK/AAB 크기**  
   3종 풀 덱 PNG는 용량이 크다. **앱 번들 분할**, 해상도별 에셋, 또는 스트리밍/온디맨드 다운로드(정책 허용 시)를 검토한다.

2. **서명**  
   스토어 배포 전 **release keystore**로 전환(현재 Gradle 설정은 문서·실제 빌드 환경에 맞게 확인).

3. **환경 분리**  
   스테이징/프로덕션 Base URL을 flavor 또는 dart-define으로 나누면 안전하다.
