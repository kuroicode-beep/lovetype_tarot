# LoveType Tarot 서버 쿨타임 QA 체크리스트

작성일: 2026-06-28  
대상: `GET /api/v1/tarot/cooltime` 서버 기준 쿨타임 전환 검증

## 01. 목적

앱이 로컬 저장소가 아니라 서버 쿨타임을 단일 진실 원천으로 사용하는지 확인한다.

## 02. 사전 조건

- debug APK 설치
- 네트워크 연결
- 사용자 프로필 저장 완료
- 가능하면 `daily`, `romance`를 각각 한 번 이상 리딩한 계정 준비

## 03. 확인 로그

debug 빌드에서는 서버 쿨타임 조회 시 다음 형태의 로그가 출력된다.

```text
[cooltime] topic=daily keys=is_available,next_available_at available=true next=null remaining=null
```

확인할 항목:

- `topic=daily`와 `topic=romance`가 각각 출력되는가
- `keys=`에 서버 응답 필드가 표시되는가
- `available=false`일 때 `next=` 또는 `remaining=`이 표시되는가
- `available=true`일 때 버튼이 활성 상태로 보이는가

## 04. 시나리오

### A. 메인 진입 직후

1. 앱 실행
2. 메인 화면 진입
3. 로그에서 `daily`, `romance` 쿨타임 조회 확인
4. 서버가 사용 가능 상태를 주면 버튼 문구가 `카드를 뽑을 수 있어요`인지 확인
5. 서버가 잠금 상태를 주면 `MM:SS 후 가능` 형식으로 표시되는지 확인

### B. 버튼 탭 시 재확인

1. 오늘의 이야기 버튼 탭
2. 로그에서 `topic=daily` 재조회 확인
3. 사용 가능이면 키워드 화면으로 이동
4. 잠금이면 안내 다이얼로그 표시

### C. 연애 이야기 포인트 흐름

1. 연애 이야기 버튼 탭
2. 로그에서 `topic=romance` 재조회 확인
3. 사용 가능이면 포인트 확인/충전 흐름 진입
4. 포인트 부족 시 충전 다이얼로그 표시

### D. 결과 생성 후 알림 예약

1. 카드 선택 후 결과 화면 진입
2. 결과 생성 성공
3. 로그에서 해당 topic 쿨타임 재조회 확인
4. `available=false`와 남은 시간이 있으면 쿨타임 해제 알림이 예약되는지 확인

## 05. 합격 기준

- 로컬 마지막 리딩 시각 없이 서버 응답만으로 잠금/활성 상태가 바뀐다.
- 서버 확인 실패 시 리딩 플로우로 진입하지 않는다.
- `daily`, `romance` 각각 독립적으로 쿨타임이 표시된다.
- 결과 생성 후 알림 시간이 고정 1시간이 아니라 서버 남은 시간 기준으로 예약된다.

## 06. 실패 시 확인 포인트

- 서버 응답 필드명이 앱 파서가 지원하는 이름과 다른지 확인
- `user_id`, `app_id`, `topic`, `category_key` 쿼리를 서버가 기대하는지 확인
- 서버가 `daily/romance` 대신 `today/love`만 받는지 확인
- 서버 시간이 UTC ISO 8601 형식인지 확인
