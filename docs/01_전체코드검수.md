## 1. 아키텍처 요약

| 구분 | 내용 |
|------|------|
| UI 프레임워크 | Flutter 3.x, Material 3 |
| 상태 | `ValueNotifier` + `ValueListenableBuilder` (`AppState`, `PaymentService`), 별도 상태관리 패키지 없음 |
| 라우팅 | `MaterialApp` `routes` + `onGenerateRoute` (`/soul-card`만 인자 전달) |
| 테마 | `AppTheme.darkTheme` / `highContrastTheme`, `AppState.highContrast`로 전환 |
| 네트워크 | `http` + 싱글톤 `ApiService` |
| 로컬 저장 | `shared_preferences` 래퍼 `StorageService` |
| 백엔드 | Railway `lovetype-api` (`AppConstants.baseUrl`) |
| 쿨타임 | 서버 `GET /api/v1/tarot/cooltime` 기준, 앱은 표시용 카운트다운만 계산 |

**평가:** 소규모 앱에 맞는 단순 구조. 화면 수가 늘어나면 라우팅·의존성 주입(go_router, get_it 등) 도입을 검토할 여지는 있음.

## 3. 라우팅·내비게이션

- **Named routes:** `/`, `/signup`, `/info-input`, `/main`, `/keyword`
- **인자 필요 화면:** `SoulCardScreen` → `/soul-card` + `arguments`
- **스택 푸시:** 카드 뽑기·확인·결과 등은 각 화면에서 `Navigator.push(MaterialPageRoute)` 직접 호출 (`app.dart` 주석과 일치)

**검수 의견:** 일관성은 유지되나, 딥링크·뒤로가기 스택 관리가 복잡해질 경우 `go_router` 등으로 정리 여지 있음.

---

## 4. 외부 연동

- **REST:** `ApiService` — GET/POST 공통 파싱, `ApiException` 처리
- **쿨타임:** `ApiService.getReadingCooltime` — 서버 응답의 `is_available`, `next_available_at`, `remaining_seconds` 계열 필드 파싱
- **인증:** Google `id_token` → `POST /auth/google` (Bearer 토큰 필드는 있으나 실사용 여부는 호출부에 따라 다름)
- **결제:** 서버 잔액·구독 조회 + IAP 영수증 연동(`IapService`) → `POST /payment/charge`
- **푸시:** FCM 등록·`NotificationService`·백그라운드 핸들러(`fcm_background.dart`)

## 9. 종합 평가

기능 단위로 모듈이 나뉘어 있고, 백엔드 스펙과의 대응 관계가 `AppConstants`·주석으로 추적 가능하다. 상태 관리가 경량인 만큼, 동시 비동기·전역 로딩이 늘어나면 `refresh`/에러 UX를 한곳에서 다루는 레이어를 두는 편이 유지보수에 유리하다.
